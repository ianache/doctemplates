# Phase 2 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

**Phase scope reminder:** This phase delivers DOCTYPE-01 and DOCTYPE-02 only — a document type list, a create form (name/description + add/remove field rows), and a detail/schema view. Per RESEARCH.md's Open Question #1, this phase ships **create + list + detail only** — no edit/delete UI or endpoints (D-03/D-04's "freely editable" is a lifecycle *policy*, not a Phase 2 build requirement; the numbered Success Criteria list only create/list/view-schema). No destructive actions exist in this phase's UI as a result. Templates, static PDFs, the visual designer, and versioning remain out of scope until Phases 3-6 — do not build nav destinations for those yet.

Revisit only when Phase 4 planning begins.

## Spacing Scale

Reuses the platform's existing scale (already declared in `frontend/tailwind.config.ts` — no new tokens needed):

| Token | Value | Usage |
|-------|-------|-------|
| base | 4px | Icon gaps, hairline offsets |
| xs | 8px | Compact spacing (badge padding, icon-to-label gaps) |
| sm | 12px | Input padding, table row vertical padding, field-row internal spacing |
| md | 16px | Default element spacing, card/panel padding, table cell padding |
| lg | 24px | Section padding, form field vertical rhythm, page content padding |
| xl | 32px | Layout gaps, gap between page header and content |
| 2xl | 48px | Major section breaks (e.g. empty-state vertical centering) |
| 3xl | 64px | Page-level spacing |

Exceptions: none. (`sm: 12px` is the one platform-wide non-4/8/16 exception, already locked in Phase 1 — not a new exception here.)

Icon-only touch targets: not needed this phase — the field-row "Remove" control is a labeled text button (not icon-only), per RESEARCH.md's code example. If a later revision makes it icon-only, use a 44px minimum target with `padding: 10px` around a 24px icon.

## Typography

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body | 14px (`body-md`, Inter) | 400 | 20px (1.43) |
| Label | 11px (`label-caps`, Inter, uppercase, +0.05em tracking) | 700 | 16px (1.45) |
| Heading | 24px (`headline-lg` size, Hanken Grotesk, -0.01em tracking) | 700 | 32px (1.33) |
| Code (field/token names) | 12px (`code-sm`, JetBrains Mono) | 400 | 18px (1.5) |

**Weight set: exactly {400, 700}.** Inter renders at 400 (Body, Code's non-mono counterparts) and 700 (Label, and Hanken Grotesk at Heading). This is the same phase-local convention Phase 1 established (`01-UI-SPEC.md`): `.design/DESIGN.md`'s platform scale defines `headline-lg` at weight 600, but this phase renders its one heading role at 700 to keep the declared weight set at 2. Do not backport this override into `.design/DESIGN.md` — it is scoped to this phase's screens, same as Phase 1's.

**Usage mapping for this phase's 3 screens:**
- Page titles ("Document Types", "New Document Type", `{document type name}`) → Heading (24px/700)
- Table column headers, form field labels, field-type badges (STRING/NUMBER/DATE/BOOLEAN) → Label (11px/700 uppercase)
- Table cell body text, form input values/placeholders, field descriptions, empty/error state body copy → Body (14px/400)
- Field/token names everywhere they appear (list row field-count is a number, not a name, so stays Body; but the detail view's actual field names, e.g. `cliente.nombre`) → Code (12px/400, JetBrains Mono) — this is the established platform convention for "Object Fields" per `.design/DESIGN.md`'s Typography section, reused here verbatim since Phase 2 is the first phase to actually render field/token names.

## Color

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `#fff8f6` (`surface` / `background`) | Page background, app shell background |
| Secondary (30%) | `#ffffff` (`surface-container-lowest`) for table/card/form surfaces; `#ffe9e6` (`surface-container`) for table header row background and left nav rail | List table container, create-form card, detail-view card, table `<thead>` row fill |
| Accent (10%) | `#610000` (`primary`) | Reserved for: the "New Document Type" / "Create Document Type" primary buttons, the "Add field" button text/icon, document type name links (list row → detail page), active left-nav item indicator, input focus border |
| Destructive | `#ba1a1a` (`error`) | Reserved for: inline validation error text (duplicate field name, invalid/missing field), the create-form's top-level error banner. No delete/destroy actions exist in this phase (see scope reminder) — do not add error-red styling to anything else. |

Accent reserved for: primary CTA buttons (New/Create Document Type, Add field), document-type-name links, active nav indicator, input focus border — never for body text, table row backgrounds, or badges (field-type badges use the neutral `surface-container` treatment from the existing document-repository mockup pattern, not accent color).

**Text-on-surface:** `#261816` (`on-surface`) for primary text (table cell values, field names' surrounding text, headings); `#5a403c` (`on-surface-variant`) for secondary/muted text (field descriptions, helper copy); `#5e5e5e` (`secondary`) for tertiary text (column headers before label-caps styling is applied, pagination/meta text, empty-state icon).

**Borders:** `#e3beb8` (`outline-variant`) for default hairlines (table row dividers, card edges, field-row separators); `#8e706b` (`outline`) for stronger borders (input borders at rest).

**Badges (field type):** Neutral treatment — `bg-surface-container text-on-surface-variant`, per the existing `.design/repositorio_documentos.html` "Type" column pattern (`<span class="bg-surface-container text-on-surface-variant px-2 py-0.5 rounded text-xs font-label-caps">`). Do NOT color-code the 4 field types individually (string/number/date/boolean) — that would spend the accent/destructive budget on non-critical decoration.

## Layout Notes (Phase 2 specific)

- **Left nav rail — introduce now, minimally.** Phase 1 deferred the full 4-destination sidebar (`Documents`/`Assets`/`Templates`/`Structure`/`Settings` from `.design/repositorio_documentos.html`) because none of those destinations existed yet. Phase 2 is the first phase with a real nav target: add a slim left nav rail (reusing the mockup's visual treatment — `w-panel-width-side` (280px), `bg-surface-container-lowest`, `border-r border-outline-variant`) containing **only** a "Document Types" item (icon: `schema`, Material Symbols — thematically matches "defining a schema"). Do not add placeholder links for Assets/Templates/Structure/Settings; add each only when its phase lands.
- **Page structure (all 3 screens):** Render inside `AuthenticatedShell`'s existing `mx-auto max-w-6xl px-lg py-2xl` content area (now via `<Outlet/>` per RESEARCH.md Pattern 3) — do NOT adopt the mockups' full-bleed 3-4 panel workspace layout; that's reserved for Phase 4's designer canvas. Page title (Heading role) at the top of each page, `xl` (32px) gap before content below it.
- **List page (`DocumentTypeListPage`):** Single data table, `surface-container-lowest` background, 1px `outline-variant` border, `rounded-lg`, per D-08. Columns in order: Name (clickable, accent-colored link → detail page), Description, Fields (count, Body text, e.g. "5 fields"), Created By, Created At. No search/filter bar this phase (unlike the document-repository mockup) — document type volume is expected to be low; add filtering only if a later phase's usage shows it's needed. "New Document Type" primary button top-right of the page header, next to the Heading title.
- **Create form (`DocumentTypeCreatePage`):** Single card (`surface-container-lowest`, 1px `outline-variant` border, `rounded-lg`, `lg` padding). Top: Name + Description text inputs (D-06). Below: field rows (D-05) — each row is Name input, Type `<select>` (string/number/date/boolean), Description input, "Remove" button — added via "Add field" button beneath the row list. Bottom: "Create Document Type" primary button, right-aligned.
- **Detail/schema view (`DocumentTypeDetailPage`):** Top section shows name (Heading), description (Body), and audit meta (Created By + Created At, Body/muted styling) — mirrors the key-value metadata block pattern in `.design/detalle_documento.html`'s left column (`flex justify-between border-b border-outline-variant/30 pb-xs` rows). Below: the field list, one row per field — field name in **Code role (JetBrains Mono)**, type as a neutral Label-styled badge, description as Body text. This reuses `.design/DESIGN.md`'s "Object Fields / Tree View" component description (mono for field identifiers) without the nested-indentation tree structure (D-02: names are opaque strings, no namespace grouping).
- **Icons:** Material Symbols Outlined only, 24px default. Nav: `schema` (Document Types). Table row link: none needed (name text itself is the link). Empty state: `folder_open` or similar neutral icon, `text-secondary`, sized larger (e.g. 48px) above the empty-state heading.
