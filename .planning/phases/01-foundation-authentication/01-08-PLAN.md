  <what-built>
    The complete AUTH-01 flow: OIDC redirect login via self-hosted Keycloak, backend-issued httpOnly session cookie, a protected authenticated shell, working Sign Out with real server-side session invalidation, and independent bearer-token validation for future API callers.
  </what-built>
  <action>
    No code changes — this task is a guided human verification pass against the live stack brought up in Task 1. Follow `<how-to-verify>` exactly and report results via `<resume-signal>`.
  </action>
  <how-to-verify>
    With the stack from Task 1 running:

    1. **Unauthenticated gating (UI path):** Open `http://localhost:5173/` in a fresh/incognito browser window with no cookies. Confirm you are redirected to `/login?error=session_expired` and see the message "Your session has expired. Please sign in again." — NOT the authenticated shell.
    2. **Login as User A:** Click "Sign In". You should be redirected to the local Keycloak login page (`localhost:8080`). Log in as `alice@example.com` / `devpass123`. Confirm you land back on `http://localhost:5173/` showing "You're signed in" with `alice@example.com` displayed in the topbar.
    3. **Inspect the cookie:** Open DevTools -> Application -> Cookies -> `http://localhost:5173`. Confirm a cookie named `docmanagement_session` (or your configured `SESSION_COOKIE_NAME`) exists, is marked `HttpOnly`, and does NOT contain a raw JWT (should be a short opaque token, not a three-part `xxx.yyy.zzz` JWT string).
    4. **Sign out:** Click "Sign Out". Confirm you land back on `/login`. Re-open DevTools -> Application -> Cookies and confirm the session cookie is gone.
    5. **Confirm real server-side invalidation:** Before signing out (redo steps 2-3 to get a fresh cookie), copy the cookie value, sign out, then manually replay a request with the old cookie value (e.g. `curl -H "Cookie: docmanagement_session=<old-value>" http://localhost:8000/api/health -i`) and confirm it returns `401` — proving the session was deleted server-side, not just cleared client-side.
    6. **Multiple distinct users:** In a second browser profile (or a second incognito window), repeat step 2 logging in as `bob@example.com` / `devpass123`. Confirm this session shows `bob@example.com` in its own topbar, independent of Alice's session (both can be active simultaneously without interfering with each other).
    7. **Login-failure error state (optional but recommended):** Temporarily stop the Keycloak container mid-flow (or navigate directly to `http://localhost:5173/login?error=callback_failed`) and confirm the copy "We couldn't sign you in." / "Something went wrong completing sign-in. Please try again, or contact your administrator if this keeps happening." / "Try Again" button renders correctly.
  </how-to-verify>
