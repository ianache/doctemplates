# Phase 1 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

**Phase scope reminder:** This phase delivers AUTH-01 only — a logged-out landing/login screen (redirect to Keycloak) and a minimal authenticated shell that proves session gating works (welcome state + sign-out). No document types, designer, content, or versioning UI exists yet — do not build nav destinations for features that don't exist until their phase lands.

## Design System

| Property | Value |
|----------|-------|
| Tool | none (no shadcn) |
| Preset | not applicable |
| Component library | none — custom Tailwind components, hand-built to match `.design/DESIGN.md` |
| Icon library | Material Symbols Outlined (Google Fonts, variable: `FILL 0, wght 400, GRAD 0, opsz 24`) |
| Font | Hanken Grotesk (headings, weights 600/700) + Inter (body/UI, weights 400/700) + JetBrains Mono (technical/token values, weight 400) |

**Source:** An existing design system named **"Precision Archival"** was found pre-built in `.design/DESIGN.md`, with working Tailwind HTML mockups (`.design/repositorio_documentos.html`, `detalle_documento.html`, `maquetacion_plantilla_documento*.html`) already implementing it. This is treated as the **canonical, locked design system for the whole platform**, not just this phase — Phase 1 must produce a login screen and authenticated shell that visually belong to the same product as those later-phase mockups.

**Discretion note (D-04 from CONTEXT.md):** RESEARCH.md recommended deferring the formal component-library pick (shadcn/ui vs. Mantine) to Phase 4, when the drag-and-drop designer's needs are concrete. Given a full custom-Tailwind token system already exists and is demonstrated in real HTML mockups (not Radix/shadcn-based), the pragmatic default for Phase 1 is: **do not initialize shadcn now**. Build the login page and shell as hand-written Tailwind components using the existing tokens below. Revisit shadcn/Mantine only if Phase 4's canvas/DnD needs make a headless primitive library worth adopting — that decision should not retroactively fight the tokens already in place. If the user disagrees, override `shadcn_initialized` before planning starts.

**Tailwind config note:** `.design/DESIGN.md` frontmatter (`rounded:` block) and the embedded `tailwind.config` script inside the `.design/*.html` mockups disagree slightly on the `borderRadius` scale (DESIGN.md: `DEFAULT 0.25rem`, `lg 0.5rem`; HTML mockups: `DEFAULT 0.125rem`, `lg 0.25rem`, `xl 0.5rem`). **DESIGN.md is the canonical source** for this contract (see Shapes below) — the mockups' embedded config appears to be an earlier draft iteration. Executor should build the real `tailwind.config` from DESIGN.md's values, not copy-paste the HTML mockup's inline script verbatim.

## Spacing Scale

Declared values (project's existing 4px-base scale, from `.design/DESIGN.md`; all multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| base | 4px | Icon gaps, hairline offsets |
| xs | 8px | Compact element spacing (icon-to-label gaps, tight padding) |
| md | 16px | Default element spacing, card/panel padding |
| lg | 24px | Section padding, form field vertical rhythm |
| xl | 32px | Layout gaps, major block separation |
| 2xl | 48px | Major section breaks (extension beyond existing scale, for this phase's centered login layout) |
| 3xl | 64px | Page-level spacing (extension beyond existing scale) |

**Exceptions:**
- `sm: 12px` — not part of the generic 4/8/16/24/32/48/64 set, but this is a **locked, already-implemented platform-wide token** from `.design/DESIGN.md`, not a per-phase judgment call. It is used for input padding and list-item padding across every existing mockup (`repositorio_documentos.html`, `detalle_documento.html`, and others). Use `sm: 12px` wherever this phase's login form inputs or any list-row padding needs it. Do **not** substitute 8px or 16px in its place — doing so would break platform-wide grid consistency with every other screen already built, and would be a worse outcome than declaring this one documented exception. No other non-standard spacing value beyond this single exception is permitted.

## Typography

Declared values (project's existing type scale, from `.design/DESIGN.md`; role mapping below is what THIS phase's login screen + shell actually use):

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body | 14px (`body-md`, Inter) | 400 | 20px (1.43) |
| Label | 11px (`label-caps`, Inter, uppercase, +0.05em tracking) | 700 | 16px (1.45) |
| Heading | 24px (`headline-lg`, Hanken Grotesk, -0.01em tracking) | 600 | 32px (1.33) |
| Display | 32px (`headline-xl`, Hanken Grotesk, -0.02em tracking) | 700 | 40px (1.25) |

**Exception — 3 weights, not 2 (locked, platform-wide, pre-existing):** This contract declares 3 distinct font weights (400, 600, 700) across the 4 roles above, exceeding the generic 2-weight-max guideline. This is **not a Phase 1 researcher judgment call** — it is a real, already-implemented cross-phase brand constraint defined in `.design/DESIGN.md` and realized in working HTML mockups that predate this phase's planning:

- `body-*` roles → Inter 400 (regular)
- `label-caps` role → Inter 700 (bold, uppercase, tracked)
- `headline-lg` / `headline-md` roles → Hanken Grotesk 600 (semibold)
- `headline-xl` role → Hanken Grotesk 700 (bold)

Because this typography system is already the platform's locked visual identity (used across `repositorio_documentos.html`, `detalle_documento.html`, and the designer mockups), Phase 1 must reuse it exactly as-is rather than collapsing to 2 weights — doing so would fragment the login/shell screens from every other screen in the product. Do not introduce a 4th weight or a 3rd font family in this phase; the exception is scoped strictly to these 3 already-declared weight values.

Supplementary size available but not needed for Phase 1 UI: `body-sm` (12px/400/16px) and `code-sm` (12px/400/18px, JetBrains Mono) — reserved for later phases' dense tabular/metadata views.

## Copywriting Contract

This phase's only screens: (1) logged-out landing/login page, (2) authenticated welcome shell, (3) login-failure error state. No document/design content exists yet — no destructive actions in this phase.

| Element | Copy |
|---------|------|
| Primary CTA (login page) | **"Sign In"** — single button, no separate "Log In" vs "Sign In" duplication. Subtext below button: "Sign in with your organization account to continue." |
| Page heading (login page) | "Precision Archival" (brand wordmark, `headline-xl`), tagline below in `label-caps` uppercase: "Document Management Platform" |
| Authenticated welcome heading | "You're signed in" |
| Authenticated welcome body | "Document management features will appear here as they're built out. Come back once document types, templates, and the designer are live." |
| Sign-out control | "Sign Out" — plain secondary button, top-right of shell topbar, no confirmation dialog (not a destructive action) |
| Error state — login callback failure | Heading: "We couldn't sign you in." Body: "Something went wrong completing sign-in. Please try again, or contact your administrator if this keeps happening." Button: "Try Again" (returns to login page) |

| Destructive confirmation | None in this phase — no delete/destroy actions exist until later phases (Phase 3+ introduces uploads, Phase 5 introduces versions). Do not add a confirmation dialog for Sign Out. |

**Language:** English for all UI chrome/copy (matches the majority convention already used in `.design/*.html` mockups — nav labels, button labels, and headings are predominantly English even though the mockups' `lang="es"` attribute and some sample data are Spanish). This is a default, not a locked upstream decision — flag to the user if Spanish-first UI is actually intended for this internal tool.

## Layout Notes (Phase 1 specific)

- **Primary visual anchor:** The "Sign In" button is the primary visual anchor on the login screen — it is the only accent-colored (`#610000`) element at rest, drawing the eye immediately. The brand wordmark ("Precision Archival," also accent-colored text but non-interactive) is the secondary anchor, establishing product identity above the CTA without competing with it for action-priority.
- **Login page:** Single centered card, ~400px wide, on the `background` (#fff8f6) page. Card styled per the existing system's "Overlay" treatment: `surface-container-lowest` background, 1px `outline-variant` border, `rounded-lg` (0.5rem per DESIGN.md), subtle shadow (4px blur). Vertical rhythm inside the card: `xl` (32px) padding, `lg` (24px) gap between wordmark block and CTA block.
- **Authenticated shell:** Minimal topbar only for this phase — brand wordmark (left), authenticated user's email + "Sign Out" button (right). Do **not** build the full left sidebar rail (Documents/Assets/Templates) shown in `.design/repositorio_documentos.html` yet — those destinations don't exist until Phase 2+ and would be dead links. Reintroduce the full sidebar shell when Phase 2 has real nav targets.
- **Icons:** Material Symbols Outlined only, 24px default size, `FILL 0 wght 400`. No other icon set.
