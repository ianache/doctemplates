# Plan 08-01: Template AST & Static Validation

This plan implements static validation of Jinja2 templates using Python-based Abstract Syntax Tree (AST) parsing. It handles nested structures, wildcard arrays, case-insensitivity, local bindings tracking, and registered globals bypassing. It enforces validation warnings for draft designs and strict rejection gates for active/superseded versions.

  <action>
    Implement a custom AST visitor JinjaTokenExtractor inheriting from jinja2.nodes.NodeVisitor in backend/app/services/content_validation.py (per D-02).
    1. Initialize self.scope_stack with an empty dictionary. Track scope bindings for target names inside visit_For, visit_Assign, visit_AssignBlock, visit_With, visit_Macro, and visit_CallBlock.
    2. Map target loop variables (e.g. item in {% for item in cliente.contactos %}) to their resolved parent path with a list wildcard suffix (e.g. cliente.contactos[]). Bind other local identifiers (or targets of unmapped loops like range(10)) to None to mark them as local block-only variables.
    3. Intercept visit_Call to only visit arguments (args/kwargs) and avoid extracting attribute methods (e.g. extracting cliente.nombre from cliente.nombre.upper()).
    4. Implement helper _get_full_path resolving Name, Getattr, Getitem (normalizing numeric indices to []), and Call (for attribute calls) using the scope stack.
    5. Skip variable validation case-insensitively if they match env.globals keys or standard Jinja2 block variables: loop, self, g, request, session (per D-03).
    6. Implement helper get_ancestor_paths(schema_path: str) -> set[str] that splits a schema field path at '.' and registers each progressive component both with and without trailing '[]' (e.g. cliente.contactos[].nombre yields {"cliente", "cliente.contactos", "cliente.contactos[]", "cliente.contactos[].nombre"}).
    7. Rewrite validate_template_tokens(html: str, allowed_tokens: list[str]) -> list[str] to run AST-based extraction and check variables against allowed ancestors. Raise 400 Bad Request if unknown variables are found, otherwise return the unique sorted extracted paths.
    8. Add extract_template_tokens_ast_warnings(html: str, valid_ancestors: set[str]) -> list[str] to check template warnings without throwing exception.

## Threat Model

### Trust Boundaries
| Boundary | Description |
|----------|-------------|
| Client → API | User templates and design status edits traverse from untrusted frontend inputs to backend database structures. |

### STRIDE Threat Register
| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-08-01 | Denial of Service | Jinja2 AST Parser | Medium | mitigate | Enforce template syntax parsing limit inside standard Python try-catch blocks, and validate document layouts on save. Mitigated in `08-01-01`. |
| T-08-02 | Information Disclosure | Jinja2 AST Extraction | High | mitigate | Track variables in local scope-stack mappings (For, Set, With, Macro) and replace local references with `None` during resolution to avoid leaking template details or registering local variables against the schema. Mitigated in `08-01-01`. |
| T-08-03 | Tampering / Bypass | Route integration status gates | High | mitigate | Strictly check design status in all PATCH/POST endpoints, preventing active designs from being updated, and strictly gate the activation route behind the AST validator. Mitigated in `08-01-02`. |
